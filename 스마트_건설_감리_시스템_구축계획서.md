## 1. 구축 개요

### 1.1 구축 목표

본 구축계획서는 사업계획서에 기반하여 **스마트 건설 감리 실시간 관리 시스템**의 실제 개발·구축을 위한 기술적 세부 사항을 정의한다.  
현장 감리 업무의 디지털 전환을 통해 보고서 자동화, 실시간 데이터 공유, 현장 안전 관리 강화를 달성하는 것을 목표로 한다.

### 1.2 구축 범위

| 구분 | 구축 대상 | 비고 |
|------|-----------|------|
| 모바일 앱 | iOS / Android 감리자용 앱 | Flutter 크로스플랫폼 |
| 웹 대시보드 | 관리자용 웹 관제 시스템 | React.js SPA |
| 백엔드 서버 | REST API 서버 | FastAPI (Python) |
| 클라우드 인프라 | AWS 기반 클라우드 환경 | 서버리스 아키텍처 활용 |
| AI 엔진 | OCR 및 음성인식 연동 | 외부 API 활용 |
| 보고서 엔진 | PDF 자동 생성 모듈 | 내부 개발 |

### 1.3 구축 원칙

- **현장 중심 설계**: 기술 친숙도가 낮은 현장 인력도 즉시 사용 가능한 직관적 UI
- **단계적 접근**: MVP 우선 완성 후 고도화 기능 순차 추가
- **확장성 확보**: 향후 현장 수 증가 및 기능 추가에 대응 가능한 아키텍처
- **데이터 무결성**: 오프라인 환경에서도 데이터 손실 없는 저장 및 동기화

### 2.2 데이터 흐름

1. **현장 → 서버**: 감리자가 앱으로 체크리스트 입력 및 사진 촬영 → API 서버 전송 → DB 저장 + S3 파일 업로드
2. **서버 → 관리자**: 데이터 저장 완료 시 WebSocket 또는 폴링으로 대시보드 실시간 갱신
3. **서버 → 감리자**: 중대 결함 등록 시 FCM/APNs 푸시 알림 발송
4. **오프라인 모드**: 통신 불량 시 로컬 SQLite에 임시 저장 → 연결 복구 시 자동 동기화

## 3. 인프라 구성 계획

### 3.1 AWS 서비스 구성

| AWS 서비스 | 용도 | 사양 (초기) |
|------------|------|-------------|
| EC2 (t3.medium) | API 서버 호스팅 | 2vCPU / 4GB RAM × 2대 (이중화) |
| RDS PostgreSQL | 관계형 데이터베이스 | db.t3.medium, 100GB SSD |
| ElastiCache (Redis) | 세션·캐시 관리 | cache.t3.micro |
| S3 | 사진·영상·PDF 저장 | 스탠다드 티어, 초기 500GB |
| CloudFront | 정적 파일 CDN 배포 | 글로벌 엣지 활용 |
| API Gateway | API 엔드포인트 관리 | REST API 유형 |
| SES | 이메일 알림 발송 | - |
| CloudWatch | 로그 및 모니터링 | - |
| WAF | 웹 방화벽 | 기본 규칙셋 적용 |

### 3.2 네트워크 구성

```
Internet
    │
    ▼
CloudFront (CDN)
    │
    ▼
ALB (Application Load Balancer)
    │
    ├── Public Subnet ──▶ EC2 (API Server #1)
    │                ──▶ EC2 (API Server #2)
    │
    └── Private Subnet ──▶ RDS (Primary)
                      ──▶ RDS (Standby - Multi-AZ)
                      ──▶ ElastiCache
```

- VPC 내 Public/Private 서브넷 분리 구성
- RDS는 Private 서브넷에 위치, 외부 직접 접근 차단
- Multi-AZ 구성으로 가용성 확보

### 3.3 확장 계획 (트래픽 증가 시)

- EC2 → Auto Scaling Group 전환
- RDS Read Replica 추가
- 필요 시 ECS (컨테이너) 기반 마이크로서비스 전환 검토

## 4. 데이터베이스 설계

### 4.1 주요 테이블 구조

#### 사용자 (users)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| id | UUID | 기본키 |
| name | VARCHAR(100) | 이름 |
| email | VARCHAR(200) | 이메일 (로그인 ID) |
| role | ENUM | admin / inspector / viewer |
| created_at | TIMESTAMP | 생성일시 |

#### 현장 (sites)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| id | UUID | 기본키 |
| name | VARCHAR(200) | 현장명 |
| address | TEXT | 주소 |
| lat / lng | DECIMAL | GPS 좌표 |
| status | ENUM | active / completed / paused |
| start_date | DATE | 공사 시작일 |
| end_date | DATE | 공사 예정 완료일 |

#### 점검 기록 (inspections)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| id | UUID | 기본키 |
| site_id | UUID | 현장 FK |
| inspector_id | UUID | 감리자 FK |
| category | VARCHAR(100) | 공종 (골조/설비/전기 등) |
| status | ENUM | pass / fail / pending |
| memo | TEXT | 음성인식 변환 메모 |
| location_lat / lng | DECIMAL | 점검 위치 |
| inspected_at | TIMESTAMP | 점검 일시 |
| is_synced | BOOLEAN | 오프라인 동기화 여부 |

#### 점검 사진 (inspection_photos)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| id | UUID | 기본키 |
| inspection_id | UUID | 점검기록 FK |
| s3_key | VARCHAR(500) | S3 저장 경로 |
| ocr_result | TEXT | OCR 인식 결과 |
| taken_at | TIMESTAMP | 촬영 일시 |

#### 결함 (defects / punch list)

| 컬럼명 | 타입 | 설명 |
|--------|------|------|
| id | UUID | 기본키 |
| inspection_id | UUID | 점검기록 FK |
| severity | ENUM | critical / major / minor |
| description | TEXT | 결함 내용 |
| resolved_at | TIMESTAMP | 조치 완료 일시 |
| resolved_by | UUID | 조치자 FK |

### 4.2 인덱스 전략

```sql
-- 현장별 점검 조회 최적화
CREATE INDEX idx_inspections_site_id ON inspections(site_id, inspected_at DESC);

-- 미동기화 데이터 조회
CREATE INDEX idx_inspections_sync ON inspections(is_synced) WHERE is_synced = false;

-- 미결 결함 조회
CREATE INDEX idx_defects_resolved ON defects(resolved_at) WHERE resolved_at IS NULL;
```

## 5. 백엔드 API 설계

### 5.1 API 엔드포인트 목록

#### 인증 (Authentication)

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | /api/v1/auth/login | 로그인 (JWT 발급) |
| POST | /api/v1/auth/refresh | 토큰 갱신 |
| POST | /api/v1/auth/logout | 로그아웃 |

#### 현장 관리

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | /api/v1/sites | 현장 목록 조회 |
| POST | /api/v1/sites | 현장 등록 |
| GET | /api/v1/sites/{id} | 현장 상세 조회 |
| PUT | /api/v1/sites/{id} | 현장 정보 수정 |

#### 점검 기록

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | /api/v1/sites/{id}/inspections | 점검 목록 조회 |
| POST | /api/v1/inspections | 점검 기록 등록 |
| POST | /api/v1/inspections/batch | 오프라인 일괄 동기화 |
| POST | /api/v1/inspections/{id}/photos | 사진 업로드 |

#### 보고서

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | /api/v1/reports/daily | 일일 감리보고서 생성 |
| POST | /api/v1/reports/weekly | 주간 감리보고서 생성 |
| GET | /api/v1/reports/{id}/download | PDF 다운로드 |

#### 대시보드

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | /api/v1/dashboard/summary | 전체 현황 요약 |
| GET | /api/v1/dashboard/defects | 미결 결함 현황 |
| WS | /ws/sites/{id}/live | 실시간 현장 데이터 스트림 |

### 5.2 API 응답 표준 형식

```json
{
  "success": true,
  "data": { ... },
  "message": "처리 완료",
  "timestamp": "2025-01-01T09:00:00Z"
}
```

오류 발생 시:

```json
{
  "success": false,
  "error": {
    "code": "INSPECTION_NOT_FOUND",
    "message": "해당 점검 기록을 찾을 수 없습니다."
  },
  "timestamp": "2025-01-01T09:00:00Z"
}
```

## 6. 프론트엔드 구성 계획

### 6.1 모바일 앱 (Flutter)

| 화면명 | 주요 기능 |

| 로그인 | 이메일/비밀번호 로그인, 자동 로그인 |

#### 오프라인 처리 전략

```
현장 데이터 입력
       │
  통신 가능?
  ┌────┴────┐
 YES       NO
  │         │
API 전송    로컬 SQLite 저장
  │         │
완료      통신 복구 감지 (Background Sync)
               │
            서버 일괄 업로드 (/batch API)
               │
            로컬 데이터 삭제
```

#### 주요 화면 구성

|--------|-----------|
| 메인 대시보드 | 전체 현장 현황, 공정률, 안전 점검 합격률 KPI |
| 현장 지도 | 현장 위치 마커, 클릭 시 현장 현황 팝업 |
| 현장 상세 | 공정별 점검 현황, 결함 목록, 담당 감리자 |
| 도면 뷰어 | 현장 도면 위 점검 위치 마커 표시 |

#### 주요 UI 컴포넌트

- **KPI 카드**: 공정률, 합격률, 미결결함 수 실시간 표시
- **Punch List 테이블**: 정렬·필터·상태 변경 인라인 처리
- **차트**: 공종별 점검 현황 (Recharts 라이브러리)
- **도면 뷰어**: 이미지 확대/축소, 마커 클릭 상세보기

## 7. 단계별 구축 일정 (WBS)
